Portfolio Redesign

Personal Project

Objective

In April of 2018, I designed and coded a website for the first time. Determined to learn more about HTML and CSS, I set out to create the whole thing from scratch, and quite honestly I fell in love the whole process of designing for the web. Even once I finished creating my portfolio website, I wanted to learn more, and chased every opportunity to keep practicing and experimenting.

Of course, six months of improving my web skills left me feeling that my initial website design was a little lacking. It's a little meta to write about on the site itself, but it needed some tuning up, so I set a few goals for the redesign:

1. Improve accessibility and general organization.

2. Alter the color scheme to something more inviting.

3. Reorganize and improve imagery, where needed.

4. Add subtle interactivity on supporting pages.

5. General updates of content, where needed.

Design

Keeping in mind that a major goal of the redesign was organization, I decided to develop the consistent or repeated elements before anything else – typography and colors. This part didn't need to be done completely from scratch; some elements from my original design were worth keeping. I especially liked the interactive gradient, body text style, and salmon/orange colors from my original design. However, all of these could use some expansion and refinement, so that is where I began.

I have always seen marked improvement after an unusual or ambitious project.

My original duo-tone color scheme got expanded to 4 major colors that would accent the salmon that was so prevalent on the site already. The new tones add a little more contrast and make everything a little less hard on the eyes. The original site only used one font family, Lato. Due to some independent exploration with condenscended sans-serifs, I decided to take the typography in a new direction and bring in Oswald and Playfair Display (sparingly) to give the typography some tasteful variety.

Once I had worked out a color/font combination that worked, it was time to venture into the less certain – accessibility and compatibility.

Cross-device compatibility was something I was really focused on when I built this website the first time, and I found a creative way to achieve a really uniform look across every possible device with the use of view-height and view-width measurements. I used them for absolutely everything. The entire page was shown relative to view-height, and while it looked great on every device, there was one big problem.

You can't zoom in on a page that's completely relative to view-height. For people who cannot see smaller text, not being able to zoom raises accessibility concerns, so I knew that in the redesign I would need to find an alternative way to have my page adapt to the screen it's on.

<meta name="viewport" content="width=device-width, initial-scale=1.0">

What a life-saver this turned out to be, at least for typography. In the original design, I had set my text to be relative to view-height, and while that generally made it readable, there remained the issue of not being able to zoom in. Using the above code in my HTML, and setting text sizes to percentages renders a comfortable size on all the devices I've tested, and bonus: you can zoom if it still isn't comfortable.

Changing how typography was handled already eliminated one reason to use relative measurements. As I designed more complicated layouts for the supporting pages of my website, it became clear that I would need a more responsive design, and a fluid grid.

"Discount Bootstrap"

Where I intern right now, I work heavily on a website with Bootstrap, and learning how to work with an existing system when designing new pages has been a unique experience to say the least. However, one of the things I have loved about the abilities Boostrap provides is its responsive column system. For my portfolio website I wanted to create a simpler version of that responsive framework.

The intial structure is pretty much the same. There are hierarchical classes of bounding divs, "container" and "row". My supporting pages have a "wrap" with a max-width, so it shows up as having margins above a certain resolution, or full-width below 1024px. Containers are 100% width. Row divs have two properties that make the inner columns work fluidly.

display:flex;

flex-wrap: wrap;

From there I developed around 10 responsive column types, notated by ID names like "span3" and "span3i". The letter "i" stands for "intermediate," indicating that those columns have a third state between the largest and smallest widths. There is also a "t" notation to indicate that the width percentage is "true" across all screen widths (useful for icons and small images). The number after span is to indicate what percentage width they are on desktop (most become 100% width on mobile). Like Bootstrap, my naming convention follows a 12 column grid, although the smallest unit my columns break into is fourths. 12 is just easier to divide by 2, 3, and 4, the main divisions I want to use on my website.

I used CSS media queries to set up 5 breakpoints, much like Bootstrap's. I took a look at UX Planet's explanation of how Bootstrap's grid works to obtain the resolutions, although I made my "top" tier a min-width of 1024px rather than 1200px to match the max-width of my supporting page container. I don't necessarily have anything changing at every breakpoint (so far), but I like having them there in case I do decide to add something someday.

Separating these things with some text, because while they don't have any margins of their own, they would normally be surrounded in things that do have margins (like this paragraph) to make them look good.
